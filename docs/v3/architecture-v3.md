# Flux v3 — Architecture

> **Flux is a deterministic execution kernel for DAG-based workflows.**
> It does not reason, it does not own users, and it does not manage conversations.
> It executes plans generated by Agent Runtime.
>
> 写作时间：2026-06-27。v3 不是 v2 的重写，而是一次**架构上的重新定位**。
> v2 的核心洞察（PlanSource 统一静态/动态、内核 definition-free、工具=MCP）全部保留，
> 并被重新划分到更清晰的五层架构中。

---

## 0. v2 → v3：什么变了、什么没变

### 核心发现

在 AgentKit 对接过程中，我们发现了 v2 架构中一直存在的**概念重叠**：

> **Agent Runtime** 和 **Workflow Runtime** 是两个不同层次的问题。
> v2 把它们放在同一个项目里，导致了两套 Agent Loop 的隐性重复。

真正重复的不是 Workflow Engine，而是 **Agent Runtime 中的 LLMPlanner**。
它实质上和 Agent Runtime 的 `Runner.RunTurn` 做的是同一件事：调 LLM → 解析 tool_call → 执行 → 回喂结果 → 循环。

同样重要的发现是：**Persistence 层从未被正式定义为架构的一层**。两套 Runtime 各自带了数据库实现（SQLite、PostgreSQL），却没有统一的 Store 接口将 Runtime 与存储解耦。

### 核心定锚：三个系统，一条依赖方向

在 v3 架构中最容易混乱的不是分层，而是 **flux、code-agent、DreamAI 三者之间的关系**。
搞清楚谁是内核、谁是宿主、谁是能力插件，是整个架构的基础：

```
DreamAI ──── 产品层（用户/多租户/业务/IAP）
    │
    │ 拥有
    ▼
Agent Runtime ──── 决策层（Loop / Tool calling / Memory / Approval）
    │
    │ 调用
    ▼
flux ──── 执行层（DAG / Scheduler / Async / Trace）
```

**一条不可违反的依赖方向：**

> **flux 不应该依赖任何 Agent。Agent 可以依赖 flux。**
> 反过来一旦成立，就会爆炸复杂度。

这意味着：
- flux 不需要"拥有 Agent 能力才能工作"。它只需要一个 `Plan`。
- 谁生成 Plan？code-agent、DreamAI Agent Runtime、Claude Code、任何 MCP client —— flux 不关心。
- DreamAI 已经是 Agent Runtime（有自己的 session / WS / tool system / user）。**不应该也不需要"集成 code-agent"**。
- code-agent 多租户升级是产品层问题，**不是 flux v3 的前置条件**。

**依赖方向的禁止清单（防回退机制）：**

| 禁止 | 理由 |
|---|---|
| ❌ Flux → Agent Runtime | flux 不知道谁在调它。引入此依赖 = 绑定特定 Agent |
| ❌ Flux → DreamAI | flux 不知道用户/产品。引入此依赖 = 退化为 DreamAI 专有引擎 |
| ❌ Agent Runtime → DreamAI internal store | Agent Runtime 是通用决策层，不绑定 DreamAI 的存储 |

**正确依赖方向（唯一允许的）：**

```
DreamAI → Agent Runtime → Flux
```

**code-agent 的定位（避免概念混淆）：**

> code-agent 是 **Agent Runtime 的一种实现**。
> 它不是标准，不是依赖，不是核心层。
> DreamAI 需要 Agent Runtime 能力，但不需要绑定 code-agent。
> DreamAI 有自己的 Agent Runtime（含 session / WS / tool system / user / IAP），它就是自己的 Agent Runtime。

**以后每次设计决策的判断标准：**

> "这是 decision 还是 execution？"
>
> | 类型 | 属于哪里 |
> |---|---|
> | 选择做什么 (decision) | Agent Runtime |
> | 如何执行 DAG (execution) | Flux |
> | 用户 / 权限 / 产品 | DreamAI |
>
> 这条规则解决所有边界争议：
> - flux 要不要 Agent 能力？ ❌ — 不是 execution
> - code-agent 要不要 DAG？ ❌ — 不是 decision
> - DreamAI 要不要 runtime？ ❌ — 不是 product

### 三种部署模式是并行关系，不是演化关系

```
                    flux kernel
                   /     |      \
            library    MCP     tool
           (DreamAI)  (任何    (code-agent
            当前方式)  Agent)   AgentKit)
```

- library、MCP、tool 不是 v1 → v2 → v3 的演化路径
- 它们是**并行的部署选项**，不同产品按需选择
- flux kernel 代码完全不变，变的是外层包装

### 变的

| | v2 | v3 |
|---|---|---|
| **flux 的定位** | Tool-First Agent Runtime | **Agent 的 Workflow Engine** |
| **Agent Loop** | flux 自己实现 `LLMPlanner` | 删掉。由 Agent Runtime 统一提供 |
| **集成方式** | flux-agent 作为独立产品 | flux 作为 Tool 接入 Agent Runtime |
| **分层模型** | 三层（编排-内核-工具） | **五层**（UI-Agent-Tool-Workflow-Persistence） |
| **持久化** | 各层自管数据库，无统一抽象 | **Store 接口层**：Runtime 只认接口，SQLite/PG 是 Adapter |

### 不变的

- `runtime.Plan` / `PlanNode` IR（definition-free）
- `PlanSource` 接口（静态/动态的统一接缝）
- `Scheduler`（依赖求解、sync/async 分叉、retry）
- `DAGPlanner`（一次性 DAG 生成 + validate → repair）
- `workflow.Compile`（DSL → Plan 编译）
- trace / replay / fork（event-sourcing 真源）
- 工具收敛到 MCP（定义层看齐 MCP，执行层保持 Flux 超集）
- `flux/runtime` 零 `flux/*` 依赖

```
                ┌──────────────────────────────────┐
                │         AgentKit UI              │  Layer 1 — UI
                │   iOS / macOS Conversation       │

                               │

           │                                       │

                                         └─────────┬─────────┘
                                                   │
                                         ┌─────────▼─────────┐
                                         │  Flux Runtime     │  Layer 4 — Workflow
                                         │  (DAG Planner +   │
                                         │   Scheduler +     │
                                         │   Workflow Engine)│
                                         └─────────┬─────────┘
                                                   │
                                         ┌─────────▼─────────┐

                                         │                   │

                                         │ WorkflowStore     │
                                         │ AwaitStore        │
                                         │ TraceStore        │

                                         │    ↓ Adapters ↓   │
                                         │ SQLite / PG /     │
                                         │ Memory / Cockroach │
                                         └───────────────────┘
```

### 五层职责

| 层 | 名称 | 职责 | 不做什么 |
|---|---|---|---|
| **Layer 1** | UI | 渲染 Conversation、展示事件流、发送消息、审批交互 | 不执行 Agent Loop、不管理会话状态 |
| **Layer 2** | Agent Runtime | Agent Loop、Conversation、Turn、Streaming、WS 协议、Tool Approval、Client Tool Execution、Reflection、Context Engineering | **不实现 Workflow Engine、不做 DAG 调度、不做 async/await** |
| **Layer 3** | Tool Layer | 注册 Tools（search/filesystem/shell/MCP/flux）、暴露 JSON Schema | 不实现 Agent Loop |
| **Layer 4** | Workflow Runtime | DAGPlanner、Scheduler、Workflow Compiler、Async/Await、Trace/Replay | **不实现 Agent Loop、不管理 Conversation/Turn/Session** |
| **Layer 5** | Persistence Ports | 定义 Store 接口。Runtime 只依赖接口，Adapter 实现具体存储 | **不定义数据库 schema、不绑定特定存储引擎** |

### 两条原则

> **任何与对话、Turn、Session、Streaming、协议相关的能力，只存在于 Agent Runtime（Layer 2）。**
> **任何与 DAG、Workflow、Scheduler、Async/Await、Trace、Replay 相关的能力，只存在于 Workflow Runtime（Layer 4）。**
> **Layer 4 不维护 Agent Loop；Layer 2 不尝试实现 Workflow Engine。**

> **任何 Runtime（Layer 2 或 Layer 4）不直接依赖 SQLite、PostgreSQL 或任何特定数据库。**
> **Runtime 只依赖 Store 接口（Layer 5），具体的 Adapter 在装配时注入。**
> **`sqlite` 不是 code-agent 的默认值；`postgres` 不是 flux 的默认值。它们只是可替换的 Adapter。**

## 3. 集成契约

### 3.1 Flux 作为 Tool 的接口

flux 的 `plan_workflow` 工具注册进 Agent Runtime 的 `tools.Registry`，实现 `tools.Tool` 接口：

```go
// Agent Runtime 侧：flux 工具包装器
type FluxWorkflowTool struct {
    engine *flux.Engine
}

func (t *FluxWorkflowTool) Name() string           { return "plan_workflow" }
func (t *FluxWorkflowTool) Description() string    { return "生成并执行一个多步 Workflow DAG..." }
func (t *FluxWorkflowTool) InputSchema() json.RawMessage { /* ... */ }

func (t *FluxWorkflowTool) Execute(ctx context.Context, ec tools.ExecutionContext, input json.RawMessage) (tools.ToolResult, error) {
    // 1. 从 input 解析 goal + 可用 tool 目录
    // 2. 调 DAGPlanner.Generate(ctx) → runtime.Plan
    // 3. 创建 MemState + Scheduler
    // 4. 调 sched.Run(ctx, NewStaticSource(plan), state)
    // 5. 等结果 / 异步通知
    // 6. 返回结果
}
```

### 3.2 Event 桥接——flux trace → Agent Runtime Emitter

flux 的 `runtime.TraceSink` 事件需要桥接到 Agent Runtime 的 `Emitter`：

```go
// flux trace sink → Agent Runtime event bridge
type bridgeTraceSink struct {
    callID string
    emit   agent.Emitter
}

func (s *bridgeTraceSink) EmitExecution(ev runtime.TraceEvent) {
    switch ev.Kind {
    case runtime.TraceNodeStart:
        s.emit(agent.Event{
            Kind:     agent.EventToolStarted,
            CallID:   s.callID,
            ToolName: ev.NodeName,
            ToolArgs: marshalJSON(ev.Payload),
        })
    case runtime.TraceOutput:
        s.emit(agent.Event{
            Kind:        agent.EventToolFinished,
            CallID:      s.callID,
            Observation: marshalJSON(ev.Payload),
        })
    }
}

func (s *bridgeTraceSink) EmitControl(ev runtime.TraceEvent) {
    // plan_extend → 可选的 observability 事件
}
```

关键原则：**桥接是单向的**。flux 不依赖 Agent Runtime 的类型，Agent Runtime 侧实现适配器。

### 3.3 Async/Await — Suspend 嵌套语义

这是最需要精确设计的部分。两个 Suspend 层次必须正确嵌套：

```
Agent Runtime Turn 生命周期
  │
  ├── Turn 开始
  │     │
  │     ├── 调 plan_workflow tool
  │     │     │
  │     │     ├── flux Scheduler.Run()
  │     │     │     ├── 同步节点执行...
  │     │     │     ├── NodeAwaiting (async 任务提交)
  │     │     │     └── 返回 StatusSuspended
  │     │     │
  │     │     ├── AwaitStore.CreateBinding(providerTaskID → node)
  │     │     └── Tool 返回 "waiting for async task <id>"
  │     │
  │     ├── Agent Runtime 继续 loop（可以干别的）
  │     │
  │     └── Turn 结束（Session 可能存档，但 AwaitBinding 独立存活）
  │
  ├── ... 时间流逝（跨 Turn / 跨 Session / 跨进程重启） ...
  │
  ├── 外部 Notify 到达
  │     │
  │     ├── AwaitStore.FindByProviderTaskID(providerTaskID) → binding
  │     ├── AwaitStore.ResolveBinding(bindingID) → claimed (原子幂等)
  │     ├── WorkflowStore.LoadPlan(taskID) → Plan (crash recovery)
  │     ├── flux Scheduler.Resume(nodeName, output)
  │     └── 继续执行 DAG 剩余节点
  │
  └── 最终结果返回
```

详细的 `AwaitBinding` 持久化契约见 [§5.2 AwaitStore](#52-store-接口定义) 和 [§5.5](#55-awaitbinding-的独立持久化suspend-嵌套的关键)。

flux v2 已有 `WithPlan` (crash recovery) 和 `PlanSource.Next` 幂等性（已知节点跳过），为跨 Turn 恢复提供了基础。

### 3.4 Tool Registry 关系

```
Agent Runtime Registry (模型可见)
  ├── read_file
  ├── grep
  ├── edit_file
  ├── run_command
  ├── web_search
  ├── plan_workflow    ← flux Tool
  └── ...

flux Registry (仅在 plan_workflow 内部可见)
  ├── generate_image   (async)
  ├── image_to_video   (async)
  ├── analyze_copy
  ├── publish
  └── merge_result
```

- **Agent Runtime 的工具**：由 Agent Runtime 的 `tools.Registry` 管理，模型直接看到
- **flux 的工具**：由 flux 的 `tool.Registry` 管理，仅当 `plan_workflow` 被调用时暴露给 DAGPlanner
- **桥接**：如果 Agent Runtime 的工具需要在 Workflow DAG 中被调用，可以通过 `plan_workflow` 的 input 传入工具定义

## 5. 持久化架构（Persistence Layer）

> 这是 v3 新增的最重要的一层。v2 从未正式定义持久化架构——两套 Runtime 各自带了数据库实现，
> 却没有统一的 Store 接口将 Runtime 与存储解耦。v3 将 Persistence 提升为架构的第一等层次。

### 5.0 核心洞察

**不应该统一数据库。应该统一接口。**

更深层的原则：**数据归属 = 生命周期所有权。谁拥有 lifecycle，谁就拥有 Store。**

| 数据类型 | 所属 | 生命周期 |
|---|---|---|
| Conversation / Turn / Message | Agent Runtime / DreamAI | 用户会话期间 |
| Workflow state / Plan / Node output | Flux | Workflow Run 期间（可能跨 Turn） |
| Await / async binding | Flux | 异步任务提交 → 回调到达 |
| User / billing / credits | DreamAI | 用户账号期间 |

```
❌ 错误思路：统一到 PostgreSQL 或统一到 SQLite

✅ 正确思路：
   Runtime  ──依赖──▶  Store 接口  ◀──实现──  SQLite Adapter
                                           ──  PostgreSQL Adapter
                                           ──  Memory Adapter (测试)
                                           ──  CockroachDB Adapter (未来)
```

不同产品在不同部署场景下选择不同存储：

| 产品 | ConversationStore | WorkflowStore | 场景 |
|---|---|---|---|
| **AgentKit (端侧)** | SQLite | SQLite | 本地 Agent，Workflow 跑在本地 |
| **DreamAI (服务端)** | PostgreSQL | PostgreSQL | 多租户，高并发 |
| **CI Agent** | SQLite / Memory | PostgreSQL | 对话短命，Workflow 需持久 |
| **SaaS (未来)** | CockroachDB | Redis + PG | 全球分布 |

Runtime 代码完全不变。变的是装配时注入哪个 Adapter。

### 5.1 两个 Aggregate，两个 Store

Agent Runtime 和 Workflow Runtime 管理的是**两个不同的 Aggregate**：

```
Conversation Aggregate          Workflow Aggregate
(Agent Runtime 拥有)             (Workflow Runtime 拥有)

Conversation                    WorkflowRun
  ├── Turn                        ├── Task
  │    ├── Message                │    ├── Node
  │    └── ToolCall               │    │    ├── NodeState
  └── Summary                     │    │    └── NodeOutput
                                  │    └── AwaitBinding
                                  └── Trace (event log)
```

它们是独立的持久化边界。一个 Agent 调用一次 Workflow = 一条 `Task` 记录，但 Conversation 不感知 Workflow 内部结构；Workflow 不依赖 Conversation 的 Turn 生命周期。

### 5.2 Store 接口定义

#### ConversationStore (Agent Runtime)

```go
// ConversationStore 是 Agent Runtime 的持久化端口。
// Agent Runtime 只依赖此接口，不依赖 SQLite/PostgreSQL/任何具体存储。
type ConversationStore interface {
    // ── Conversation 生命周期 ──
    CreateConversation(ctx context.Context, meta ConversationMeta) (*Conversation, error)
    LoadConversation(ctx context.Context, conversationID string) (*Conversation, error)
    ListConversations(ctx context.Context, opts ListOptions) ([]ConversationSummary, error)
    DeleteConversation(ctx context.Context, conversationID string) error

    // ── Message / Turn ──
    AppendMessage(ctx context.Context, conversationID string, msg Message) error
    AppendMessages(ctx context.Context, conversationID string, msgs []Message) error

    // ── Compaction ──
    SaveSummary(ctx context.Context, conversationID string, summary string, droppedMessageCount int) error

    // ── Event (append-only, 用于 replay) ──
    AppendEvent(ctx context.Context, conversationID string, ev EventRecord) error
    ReplayEvents(ctx context.Context, conversationID string, sinceSeq int64) ([]EventRecord, error)
}

// ConversationMeta 是创建会话时的元数据（非对话内容）。
type ConversationMeta struct {
    ID            string
    WorkspacePath string
    Model         string
    Title         string // 从第一条 user message 提取
}

// Conversation 是 Agent Runtime 操作的会话对象。
type Conversation struct {
    Meta     ConversationMeta
    Messages []Message
    Summary  string
    // PromptTokens / ContextWindow / CompactThreshold 等
}
```

#### WorkflowStore (Workflow Runtime)

```go
// WorkflowStore 是 Workflow Runtime 的持久化端口。
// flux 的 Engine/Scheduler 只依赖此接口。
type WorkflowStore interface {
    // ── WorkflowRun ──
    CreateRun(ctx context.Context, meta RunMeta) (*RunID, error)
    LoadRun(ctx context.Context, runID string) (*WorkflowRun, error)

    // ── Task ──
    CreateTask(ctx context.Context, runID string, meta TaskMeta) (*TaskID, error)
    LoadTask(ctx context.Context, taskID string) (*Task, error)
    ListTasks(ctx context.Context, runID string) ([]Task, error)

    // ── Node ──
    UpdateNodeState(ctx context.Context, taskID string, nodeName string, state NodeState, output map[string]any) error
    LoadNodeStates(ctx context.Context, taskID string) (map[string]NodeState, error)

    // ── Plan 持久化（crash recovery） ──
    SavePlan(ctx context.Context, taskID string, plan *Plan) error
    LoadPlan(ctx context.Context, taskID string) (*Plan, error) // nil 表示无已保存 plan
}

// RunMeta 是 Workflow 运行的元数据。
type RunMeta struct {
    ID            string // 外部传入或生成
    ConversationID string // 调起此 workflow 的 Conversation（关联，非依赖）
    Goal          string
    ToolCatalog   []ToolDefinition // DAGPlanner 的工具目录快照（回放时需要）
}

// TaskMeta 是一次任务执行的元数据。
type TaskMeta struct {
    ID       string
    ParentID string // 子 task 的父 task
    RootID   string // 根 task
}
```

#### AwaitStore (Workflow Runtime — 异步等待)

```go
// AwaitStore 管理 async 节点的挂起/恢复凭证。
// 它必须独立于 Turn 生命周期持久化——Turn 结束不代表 Workflow 结束。
type AwaitStore interface {
    // CreateBinding 登记一个异步等待。
    CreateBinding(ctx context.Context, binding AwaitBinding) error

    // ResolveBinding 原子完成一个等待（幂等：重复 resolve 返回 false）。
    ResolveBinding(ctx context.Context, bindingID string) (claimed bool, err error)

    // FindByProviderTaskID 通过外部 provider task ID 查找 binding。
    FindByProviderTaskID(ctx context.Context, providerTaskID string) (*AwaitBinding, error)

    // ListPending 列出所有 awaiting 状态的 binding（用于 poll worker 扫描）。
    ListPending(ctx context.Context) ([]AwaitBinding, error)
}

type AwaitBinding struct {
    BindingID      string
    TaskID         string
    NodeName       string
    ProviderTaskID string    // 外部 Provider 返回的 job_id / task_id
    Status         string    // "awaiting" | "completed" | "failed"
    CreatedAt      time.Time
    ResolvedAt     time.Time // zero = 未完成
}
```

#### TraceStore (Workflow Runtime — event sourcing)

```go
// TraceStore 是 trace event log 的持久化端口。
// trace 是确定性/回放/分叉的真源。
type TraceStore interface {
    AppendTrace(ctx context.Context, taskID string, events []TraceEvent) error
    ReplayTrace(ctx context.Context, taskID string, sinceSeq int64) ([]TraceEvent, error)
}
```

### 5.3 Store 不绑定数据库——Adapters

```
                                  ┌──────────────────────┐
                                  │   ConversationStore  │  ← 接口 (Layer 5)
                                  │   (interface)        │
                                  └──┬──────┬──────┬─────┘
                                     │      │      │
                            ┌────────▼┐ ┌───▼───┐ ┌▼──────────┐
                            │ SQLite  │ │  PG   │ │  Memory   │  ← Adapters
                            │ Conv.   │ │ Conv. │ │  Conv.    │
                            │ Store   │ │ Store │ │  Store    │
                            └─────────┘ └───────┘ └───────────┘

                                  ┌──────────────────────┐
                                  │   WorkflowStore      │  ← 接口 (Layer 5)
                                  │   (interface)        │
                                  └──┬──────┬──────┬─────┘
                                     │      │      │
                            ┌────────▼┐ ┌───▼───┐ ┌▼──────────┐
                            │ SQLite  │ │  PG   │ │  Memory   │  ← Adapters
                            │ WF      │ │ WF    │ │  WF       │
                            │ Store   │ │ Store │ │  Store    │
                            └─────────┘ └───────┘ └───────────┘
```

每个 Adapter 是一个独立的包：

```
Adapter 目录结构（示意）：

flux/adapter/
  sqlite/
    conversation_store.go   // SQLiteConversationStore
    workflow_store.go        // SQLiteWorkflowStore
    await_store.go           // SQLiteAwaitStore
    trace_store.go           // SQLiteTraceStore
  postgres/
    conversation_store.go    // PGConversationStore
    workflow_store.go        // PGWorkflowStore
    await_store.go           // PGAwaitStore
    trace_store.go           // PGTraceStore
  memory/
    conversation_store.go    // MemoryConversationStore (测试用)
    workflow_store.go        // MemoryWorkflowStore (测试用)
```

**Adapter 不引入新依赖到 Runtime。** `flux/runtime` 不 import 任何 adapter 包。Adapter 是纯实现，依赖 Runtime 的类型（`Plan`、`NodeState`、`AwaitBinding` 等），但 Runtime 的类型不引用 Adapter。

### 5.4 装配示例

不同产品在启动时组装不同的 Adapter：

```go
// AgentKit (端侧) — 全部 SQLite
func setupAgentKit() {
    convStore := sqlite.NewConversationStore("~/.agentkit/sessions.db")
    wfStore   := sqlite.NewWorkflowStore("~/.agentkit/workflows.db")
    awaitStore := sqlite.NewAwaitStore("~/.agentkit/workflows.db")

    agentRT := agent.NewRuntime(agent.Config{
        ConversationStore: convStore,
        // ... 其他配置
    })
    wfEngine := flux.NewEngine(flux.Config{
        WorkflowStore: wfStore,
        AwaitStore:    awaitStore,
        // ... 其他配置
    })
    agentRT.Tools.Register(flux.NewWorkflowTool(wfEngine))
}

// DreamAI (服务端) — 全部 PostgreSQL
func setupDreamAI() {
    pool := pgxpool.Connect("postgres://...")
    convStore := postgres.NewConversationStore(pool)
    wfStore   := postgres.NewWorkflowStore(pool)
    awaitStore := postgres.NewAwaitStore(pool)

    agentRT := agent.NewRuntime(agent.Config{
        ConversationStore: convStore,
    })
    wfEngine := flux.NewEngine(flux.Config{
        WorkflowStore: wfStore,
        AwaitStore:    awaitStore,
    })
    agentRT.Tools.Register(flux.NewWorkflowTool(wfEngine))
}

// 测试 — 全部 Memory
func setupTest(t *testing.T) {
    convStore := memory.NewConversationStore()
    wfStore   := memory.NewWorkflowStore()
    // ...
}
```

关键：`agent.NewRuntime` 和 `flux.NewEngine` 的签名在三个场景中**完全相同**。
变的只是注入的 Adapter 实现。

### 5.5 AwaitBinding 的独立持久化——Suspend 嵌套的关键

这是 Persistence Layer 设计中最需要精确的部分（见 §3.3 的 Suspend 嵌套语义）：

```
时间线：
  Turn 开始 ─────────────────────────── Turn 结束 ────── Notify 到达 ───── Resume
  │                                       │                 │               │
  │  plan_workflow.Execute()              │                 │               │
  │    └─ Scheduler.Run()                 │                 │               │
  │         └─ NodeAwaiting               │                 │               │
  │              └─ AwaitStore.           │                 │               │
  │                 CreateBinding() ──── 持久化 ───────── 跨 Turn 存活 ──→ FindByProviderTaskID()
  │                                       │                 │               │
  └─ Tool 返回 "waiting..."               │                 │               │
                                          │                 │               │
                                   Session 保存      外部回调到达      Resume 执行
                                  (ConversationStore)  (AwaitStore)   (WorkflowStore)
```

`AwaitBinding` 的持久化**不能依赖 Conversation 的生命周期**。它必须独立存储，在 Conversation Turn 结束、甚至 Session 被删除后仍然存活。这就是为什么 `AwaitStore` 是 Workflow Runtime 的独立端口，不属于 ConversationStore。

### 5.6 设计决策

| 决策 | 结论 | 理由 |
|---|---|---|
| 统一数据库还是统一接口？ | **统一接口** | SQLite 适合端侧；PG 适合服务端。统一数据库 = 限制场景 |
| Conversation 和 Workflow 是同一个 Store 吗？ | **两个独立 Store** | 两个 Aggregate，不同的持久化边界，不同的生命周期 |
| AwaitBinding 存在哪个 Store？ | **AwaitStore（独立）** | 必须跨 Turn 存活，不属于 Conversation 的生命周期 |
| Store 接口放在哪里？ | **Runtime 侧定义接口，Adapter 侧实现** | `flux/runtime` 定义 `WorkflowStore` 接口；`flux/adapter/` 实现 |
| Adapter 是否引入数据库依赖到 runtime？ | **不** | `runtime/` 零外部依赖。Adapter import runtime 的类型，反之不可 |
| 这个模式只适用于 flux 吗？ | **不。对称的** | code-agent 的 ConversationStore 应该走同样的接口+adapter 模式。Runtime 定义接口，消费者（AgentKit/DreamAI）提供实现。两边遵循同一模式（见 §8.3） |

## 6. 设计原则（不可被重新争论）

这些是从 v2 继承并经 v3 强化的原则：

### 6.0 决策 vs 执行——最高优先级判断标准（v3 新增）

> 以后每次问"这个要不要进 flux？"，只问一句：
> **"这是 execution 问题，还是 decision 问题？"**
>
> | 类型 | 放哪里 | 例子 |
> |---|---|---|
> | 选择做什么（decision） | Agent Runtime | 调哪个工具、下一步做什么、是否重试 |
> | 怎么执行 DAG（execution） | flux | 依赖求解、并行调度、async await、trace replay |
> | 用户/权限/产品 | DreamAI / 宿主 | 多租户、IAP、credits、UI |

### 6.1 机制 vs 编排（保留）

> 内核（kernel）只认"机制"（依赖、并行、异步等待、状态、trace、回放）；
> "编排概念"（LLM planner、DSL 编译、Agent Loop）在上层。

v3 强化：Agent Loop 完全上移到 Agent Runtime；flux kernel 只认 `PlanSource` → `Plan` → `Scheduler`。

### 6.2 state system vs event system（保留）

> `ExecState` 是当前状态快照；trace 是历史真相演化。
> trace = event log，是三性的唯一真源。

### 6.3 工具 = MCP（保留）

> 定义层向 MCP 看齐（JSON Schema / outputSchema），执行层保持 Flux 超集（async/await/事件）。
> MCP 是接缝后的一种 binding，不是接口本身。

### 6.4 Agent Runtime 与 Workflow Runtime 分离（v3 新增）

> 任何与 Conversation/Turn/Session/Streaming/Protocol 相关的能力，只存在于 Agent Runtime。
> 任何与 DAG/Scheduler/Async-Await/Trace-Replay 相关的能力，只存在于 Workflow Runtime。
> 二者通过 Tool Invocation 集成，互不内嵌。
>
> **依赖方向不可逆**：flux 不应该依赖任何 Agent。Agent 可以依赖 flux。
> 反过来一旦成立，就会爆炸复杂度。

### 6.5 Flux 是 Agent 的 Workflow Engine，不是某个 Agent 的专属引擎（v3 新增）

> Flux 不绑定任何特定 Agent（DreamAI、Code Agent、Research Agent...）。
> 任何 compliant Agent 都可以通过 Tool 协议调用 Flux。
> Flux 不知道自己被谁调用——它只知道收到一个 goal + tool catalog，产出一个执行结果。

### 6.6 runtime 不 import flux（保留+强化）

> `flux/runtime` 保持零 `flux/*` 依赖，纯 stdlib。
> 这是 flux 可以被独立编译、独立嵌入、独立测试的前提。

### 6.7 Persistence 是端口，不是数据库（v3 新增）

> Runtime（Agent Runtime 或 Workflow Runtime）不直接依赖 SQLite、PostgreSQL 或任何特定存储引擎。
> Runtime 只依赖 Store 接口。SQLite/PostgreSQL/Memory 是 Store 接口的 Adapter 实现。
> 产品在装配时选择 Adapter；Runtime 代码不变。
> 两个 Aggregate（Conversation 和 Workflow）拥有独立的 Store 接口和独立的持久化边界。

### 6.8 Flux 不强迫调用方采用特定 Agent Runtime（v3 新增）

> Flux 支持三种集成模式：嵌入式库（Go import）、MCP 服务（标准协议）、Tool 嵌入（同进程 Tool 接口）。
> 调用方可以选择任何模式，也可以同时使用多种模式。
> Flux 不要求调用方使用 code-agent 或任何特定 Agent Runtime。
> 当前使用模式 A（嵌入式库）的调用方（如 DreamAI）可以零迁移成本继续使用。

## 8. 集成模式与部署拓扑

### 8.0 不要过早耦合

v3 架构描述的是**长期方向**，不是**立即重构**。特别是以下两点需要诚实面对：

1. **code-agent 当前是单机/单用户设计**。它没有用户系统、多租户隔离、权限模型。让 DreamAI 直接嵌入 code-agent 意味着需要先完成这些能力——这是一个独立的大项目，不是 flux v3 的前置条件。
2. **DreamAI 已经有自己的会话管理、WebSocket、用户系统**。强行用 code-agent 替换这一层，ROI 可能很低。

因此，v3 的核心原则中有一条隐式的约束：**flux 不应强迫任何 Agent 采用特定的 Runtime**。flux 应该能在多种部署模式下工作，从"嵌入 Go 库"到"独立 MCP 服务"。

### 8.1 三种部署模式

Flux v3 支持三种集成模式。**它们不是演化关系（library → MCP → tool），而是并行的部署选项。**
同一份 flux kernel 代码完全不变，变的是外层包装。不同产品按需选择。

```
模式 A: 嵌入式库（当前 DreamAI 的方式）
─────────────────────────────────────────
DreamAI Go 进程
  ├── 自己的 Agent Runtime (Conversation/WS/User)
  ├── 自己的 Tool Registry
  └── import "flux/engine"
        ├── DAGPlanner
        ├── Scheduler
        └── WorkflowStore (PostgreSQL)

→ 零网络开销
→ DreamAI 保持对自己的 Agent Runtime 的完全控制
→ flux 只是被调用的 Go 库，和现在一样


模式 B: MCP 服务（标准协议集成）
─────────────────────────────────────────
DreamAI / Claude Code / 任何 MCP Agent
  │
  │ MCP 协议 (stdio / SSE / streamable HTTP)
  │
  ▼
flux-mcp-server (独立进程)
  ├── DAGPlanner
  ├── Scheduler
  └── WorkflowStore + AwaitStore

→ 语言无关：Python/JS/Rust Agent 都能调
→ 任何支持 MCP 的 Agent (Claude Code、Codex、自研) 直接可用
→ flux 对 Agent 完全透明——Agent 只知道一个 MCP tool: plan_workflow


模式 C: Tool 嵌入（代码级同进程）
─────────────────────────────────────────
Agent Runtime Go 进程 (code-agent 或任何 Go Agent)
  ├── tools.Registry
  │     └── FluxWorkflowTool (实现 tools.Tool)
  │           └── import "flux/engine"
  └── 与 Agent Loop 同进程

→ 最低延迟
→ 共享 Tool Registry、Emitter、Context
→ flux trace 可桥接为 Agent Runtime Event
```

**三种模式的关键差异：**

| | 模式 A (嵌入式库) | 模式 B (MCP 服务) | 模式 C (Tool 嵌入) |
|---|---|---|---|
| **谁持有 Agent Runtime** | 宿主进程自己 | 调用方 Agent | Agent Runtime |
| **flux 的接口** | Go API (`Engine.Run/Notify`) | MCP `tools/call` | `tools.Tool.Execute()` |
| **部署形态** | 同进程 | 独立进程/服务 | 同进程 |
| **适用场景** | DreamAI (已有 Agent) | Claude Code、多语言 Agent | code-agent / AgentKit |
| **迁移成本** | **无**。就是当前方式 | 较小的包装层 | 中等 |
| **Agent Runtime 依赖** | 无。flux 不依赖 Agent Runtime | 无。纯 MCP 协议 | 依赖 Agent Runtime 的 Tool 接口 |

### 8.2 DreamAI 的迁移路径（务实版）

**核心原则：DreamAI 已经是 Agent Runtime。不应该也不需要"集成 code-agent"。**

当前状态：DreamAI 嵌入 flux 作为库（模式 A），有自己的完整 Agent Runtime（session / WS / tool system / user / IAP）。

**v3 对 DreamAI 的建议路径：**

```
第一优先级：LLMPlanner 删除 → 不影响 DreamAI
─────────────────────────────────────────────
DreamAI 从未使用过 LLMPlanner（那是 flux-agent CLI 用的）。
删除 LLMPlanner 对 DreamAI 零影响。立即可以做。


第二优先级：Store 接口抽象 → DreamAI 直接受益
─────────────────────────────────────────────
当前 flux 的数据访问耦合在 engine 里。
抽象 WorkflowStore / AwaitStore 后，DreamAI 注入自己的 PG 连接池。
flux 变成更干净的库。


第三优先级：MCP 服务暴露 → 可选叠加
─────────────────────────────────────────────
flux-mcp-server 让 flux 可以独立部署。
DreamAI 可同时用模式 A（库，低延迟）+ 模式 B（MCP，独立扩展）。
两种模式共存，渐进切换。


不需要做的事
─────────────────────────────────────────────
❌ 让 DreamAI "集成 code-agent"
   → DreamAI 已有 Agent Runtime。引入 code-agent = 双 Runtime 冲突
   → 两套 session、两套 memory、两套 tool loop、两套 event system

❌ 等 code-agent 完成多租户再推进 flux v3
   → code-agent 多租户是产品层问题，flux 是系统层问题
   → 互不阻塞
```

### 8.3 code-agent 的多租户升级——独立轨道

code-agent 当前状态：

| 能力 | 现状 | 多租户需要 |
|---|---|---|
| 数据库 | SQLite 单文件 | 接口化 → SQLite / PG / MySQL adapter |
| 会话隔离 | 单用户，单 workspace | 用户 ID 隔离 + 权限 |
| WS 连接 | 单一 RemoteApprover | 按用户路由 |
| Auth | 无 | 认证 + 授权 |

**协议化是对称的。** flux 的 WorkflowStore/AwaitStore 走接口+adapter 模式（§5），code-agent 的 ConversationStore 应该走同样的模式：

```
code-agent 定义接口               消费者提供实现
─────────────────────           ─────────────────
ConversationStore               SQLiteConversationStore  (AgentKit 端侧)
  (interface)                   PGConversationStore      (DreamAI 服务端)
                                MemoryConversationStore  (测试)
```

这样 DreamAI 不需要"集成 code-agent"，而是提供自己的 `PGConversationStore` 实现，注入到 code-agent Runtime。数据归 DreamAI，Runtime 归 code-agent。和 flux 的 Store 抽象完全对称。

**这不在 flux v3 的范围内。** 这是 code-agent 自己的演进路线，和 flux 的架构调整是两个独立项目。此处记录是为了表明：两边遵循同一个模式，未来对接时不会有架构冲突。

### 8.4 为什么架构仍然要先画出来

尽管有上述现实约束，架构文档仍然有价值：

1. **Store 接口抽象** 对 DreamAI 直接受益——当前 flux 的数据库访问耦合在 engine 里，抽象后 DreamAI 可以注入自己的 PG 连接池
2. **MCP 暴露** 让 flux 从 "DreamAI 的库" 变成 "DreamAI 的 MCP 服务"——部署、扩展、监控全部独立
3. **LLMPlanner 删除** 消除代码负担——一个 DreamAI 从不使用的模块
4. **Agent Runtime / Workflow Runtime 分离** 阻止未来继续在 flux 里长 Agent 能力

## 附录 A：决策记录

### D1 — 为什么删 LLMPlanner 而不是 Agent Runtime Runner

1. **能力覆盖**：Agent Runtime 的 Runner 是 LLMPlanner 的超集（Streaming、Approval、Client Tool、Reflection、Compaction 全部缺失于 LLMPlanner）
2. **维护成本**：LLMPlanner 已经是事实上的"第二套 Agent Runtime"，长期维护意味着两边的 Conversation/Turn/Protocol 都要同步演进
3. **关注点分离**：LLMPlanner 让 flux 负担了不属于 Workflow Engine 的职责。删掉它让 flux 回到核心使命

### D2 — 为什么保留 DAGPlanner

1. **它是 Workflow 语义的 LLM 入口**：DAGPlanner 产出的不是 agent 的增量决策，而是一张完整的、可校验的 DAG
2. **generate → validate → repair 循环**（FR5）是 Workflow Engine 独有的能力，Agent Runtime 不需要也不应该实现
3. **它是 PlanSource 的一个实现**：不影响内核，不引入 Agent Loop 语义

### D3 — 为什么 flux/runtime 保持零依赖

这是 flux 能被任何 Agent 嵌入的前提。如果 runtime 依赖了任何外部包（包括但不限于某个 Agent Runtime 的类型），那它就又绑定回一个特定产品了。

### D4 — 为什么用 Tool 协议集成，而不是库依赖

1. **Tool 是 Agent Runtime 最成熟的扩展点**：Agent Runtime 已经有 `ClientTool`、`SideEffecting`、`SkillAnnouncer` 等接口模式，加一个 `plan_workflow` tool 不违反任何设计原则
2. **库依赖会制造循环**：flux 需要知道 Agent 的上下文；Agent 需要知道 flux 的执行状态。Tool 协议天然解耦
3. **多 Agent 共享**：Tool 协议是 Agent Wire Protocol 的一部分，任何实现该协议的 Agent 都能调用 flux

### D5 — 为什么不统一数据库，而是统一接口

1. **部署场景决定存储**：端侧用 SQLite（零运维），服务端用 PostgreSQL（高并发）。统一数据库 = 强制所有场景适配一种存储
2. **数据库是 Adapter，不是核心**：`ConversationStore` 和 `WorkflowStore` 的接口语义不依赖存储引擎。SQLite 和 PostgreSQL 只是两种实现
3. **测试友好**：Memory Adapter 让 Runtime 的集成测试不需要任何外部依赖
4. **未来可扩展**：CockroachDB、DynamoDB、Redis —— 只要实现 Store 接口即可
5. **关键约束**：Adapter 不引入新依赖到 Runtime。`runtime/` 既不 import `sqlite` 也不 import `pgx`

### D6 — 为什么 AwaitBinding 必须独立于 Conversation 持久化

1. **生命周期不同**：一个 Workflow Run 可能跨越多个 Conversation Turn，甚至跨越多个 Session。AwaitBinding 的存活期 ≥ Workflow Run > Turn
2. **所有权不同**：`AwaitBinding` 属于 Workflow Runtime（`NodeAwaiting` 状态），不属于 Agent Runtime
3. **Notify 不依赖 Conversation**：外部 Provider 回调到达时，Conversation 可能已经不存在（用户删掉了 session）
4. **独立存储 = 独立恢复**：`AwaitStore` 可以有自己的 poll worker 扫描 awaiting 状态，不依赖 Conversation 的活跃度

### D7 — 为什么 flux 支持三种部署模式而不是只选一种

1. **模式 A (嵌入式库) 不能丢**：DreamAI 当前就是这样用 flux。破坏这个模式 = 强迫 DreamAI 做无谓的迁移
2. **模式 B (MCP 服务) 打开最大市场**：Claude Code、Codex、任何 MCP Agent 都能用。flux 从"DreamAI 的引擎"变成"Agent 生态的引擎"
3. **模式 C (Tool 嵌入) 给最深集成**：AgentKit/code-agent 场景下，同进程调用 + Event 桥接 = 最低延迟 + 最佳体验
4. **三种模式共享同一份 flux 代码**：DAGPlanner、Scheduler、WorkflowStore 不变。变的只是外层包装（MCP server / Tool adapter / 直接 import）
5. **迁移不必一步到位**：DreamAI 可以从模式 A 开始，渐进加模式 B（MCP 暴露），再评估是否切到 Store 接口
